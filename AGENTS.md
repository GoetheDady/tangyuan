# 汤圆 Agent 说明

## 基本约定

- **语言**：始终使用中文回复，包括解释、计划、commit message、PR 描述以及写入文档的内容。代码、标识符、API 名称保持英文。
- **查代码优先用 codegraph**：仓库已建有 `.codegraph/` 索引。理解或定位代码时，先用 `codegraph explore <query>` 或 `codegraph query <search>`，需要调用关系时用 `codegraph callers/callees/impact`，再回退到 `rg` 等文本搜索。

## 行为准则

用于减少常见 LLM 编码错误的行为准则。与上面的项目专属说明结合使用。

**权衡取舍：** 这些准则偏向谨慎而非速度。对于琐碎任务，请自行判断。

### 1. 先思考，再编码

**不要假设。不要隐藏困惑。暴露权衡取舍。**

实现之前：

- 明确说明你的假设。如果不确定，就提问。
- 如果存在多种解读，把它们都列出来，不要默默选一个。
- 如果有更简单的方案，就说出来。有理由时要提出反对意见。
- 如果有不清楚的地方，停下来。指出困惑点，然后提问。

### 2. 简单优先

**用最少的代码解决问题。不做任何推测性的东西。**

- 不实现超出需求的功能。
- 不为一次性代码做抽象。
- 不加入未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景做错误处理。
- 如果你写了 200 行而它本可以是 50 行，重写它。

问自己一句：“资深工程师会觉得这写得过于复杂吗？”如果是，就简化。

### 3. 外科手术式修改

**只动你必须动的地方。只清理你自己制造的混乱。**

修改现有代码时：

- 不要“顺手改进”相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即便你会用不同的写法。
- 如果发现无关的死代码，提一句就行，不要删。

当你的改动产生了孤儿代码时：

- 移除因你的改动而变得无用的 import、变量、函数。
- 除非被要求，不要移除原本就存在的死代码。

检验标准：每一行改动都应能直接追溯到用户的请求。

### 4. 目标驱动的执行

**定义成功标准。循环直到验证通过。**

把任务转化为可验证的目标：

- “加个校验” → “为非法输入写测试，然后让它们通过”
- “修复这个 bug” → “写一个能复现它的测试，然后让它通过”
- “重构 X” → “确保重构前后测试都通过”

对于多步骤任务，先写一个简短的计划：

```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强的成功标准让你能独立循环推进。弱的标准（“让它能跑”）会导致不断需要澄清。

---

**这些准则是否奏效，看这些迹象：** diff 里不必要的改动更少了，因过度复杂而返工的情况更少了，澄清性问题出现在实现之前而非犯错之后。

## 开发规范

### 文件长度

- 单个源文件不超过 **500 行**为宜，超过 1000 行必须拆分。
- 拆分策略：按职责将类型定义、schemas、核心逻辑、工具函数等分到独立文件中，通过 barrel export (`index.ts`) 统一对外暴露。
- 测试文件同理，与源文件保持相近的粒度。

### 技术栈

- 语言：TypeScript（strict 模式）
- 运行时：Node.js（Main 进程）/ Electron + React（Renderer）
- 验证库：Zod（contracts 包）
- 测试框架：Vitest
- Monorepo 工具：pnpm workspaces

## Agent skills

### Issue tracker

Issues 和 PRD 使用 GitHub Issues，仓库是 `git@github.com:GoetheDady/tangyuan.git`；外部 PR 不作为 triage 请求入口。详见 `docs/agents/issue-tracker.md`。

### Triage labels

Triage 角色映射为中文 GitHub 标签：`待评估`、`需要更多信息`、`可交给 Agent`、`需要人工处理`、`不处理`。详见 `docs/agents/triage-labels.md`。

### Domain docs

本仓库使用 single-context 领域文档结构：根目录 `CONTEXT.md` 加 `docs/adr/`。详见 `docs/agents/domain.md`。
